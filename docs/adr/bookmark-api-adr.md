# ADR: bookmark-api アーキテクチャ決定記録

## ADR-001: Go + net/http 標準ライブラリの採用

**ステータス**: 承認済み

**コンテキスト**: APIフレームワークとして gin, echo, chi などの選択肢がある。

**決定**: Go 1.22+ の `net/http` 標準 `ServeMux` を使用する。

**理由**:
- Go 1.22 で `ServeMux` にメソッドベースルーティングとパスパラメータが追加され、この規模のAPIには十分
- 外部依存ゼロで保守コストが低い
- 標準ライブラリなのでGoのバージョンアップに追従すれば壊れない

**トレードオフ**: ミドルウェアチェーンの仕組みがない。ロギング等が必要になったら手動で関数を重ねるか、その時点でchiへの移行を検討。

## ADR-002: インメモリストアをinterface経由で利用

**ステータス**: 承認済み

**コンテキスト**: 要件上DBは不要だが、将来DB対応する可能性がゼロではない。

**決定**: `Store` interfaceを定義し、`MemoryStore` で実装する。handlerはinterfaceに依存する。

**理由**:
- interface定義のコストはGoでは極めて低い（メソッド3つ）
- テスト時にモック差し替えが容易
- DB移行時にhandler側の変更が不要

**トレードオフ**: 「YAGNI違反では？」→ interfaceの定義コストが十分低く、テスタビリティの恩恵が即座にあるため正当化できる。

## ADR-003: 3パッケージ構成（handler / model / store）

**ステータス**: 承認済み

**コンテキスト**: Go プロジェクトの構成パターンとして、フラット構成、レイヤード構成、DDD的構成がある。

**決定**: handler / model / store の3パッケージ + main。

**理由**:
- エンドポイント3つのAPIに対して、service層やrepository層を分けるのは過剰
- handler がストアを直接呼ぶシンプルな構造
- パッケージ境界が明確で、各パッケージのテストが独立可能

**トレードオフ**: ビジネスロジックが複雑化した場合、handlerが肥大化する。その時点で `service` パッケージを導入すればよい。

## ADR-004: タグフィルタのAND条件

**ステータス**: 承認済み

**コンテキスト**: `GET /bookmarks?tag=go&tag=tech` で複数タグ指定時の挙動を決める必要がある。

**決定**: AND条件（すべてのタグを持つブックマークのみ返す）。

**理由**:
- 「goかつtechのブックマーク」という絞り込みが自然なユースケース
- 実装が単純（全タグの包含チェック）
- OR条件が必要になったら `tag_or` パラメータを追加可能

**トレードオフ**: OR条件が必要な場合はAPIの追加変更が必要。ただし、既存の `tag` パラメータの意味を変えずに拡張できるため、後方互換性は保てる。

## ADR-005: IDにUUID v4を採用

**ステータス**: 承認済み

**コンテキスト**: ブックマークの識別子として連番ID、UUID、nanoidなどの選択肢がある。

**決定**: UUID v4 を採用。Go標準の `crypto/rand` ベースで生成するか、`google/uuid` パッケージを使用。

**理由**:
- 連番はストアに採番カウンターが必要で、並行処理時にロック設計が複雑化する
- UUIDはステートレスに生成可能
- URLパスに含めても安全な文字列

**トレードオフ**: IDが長い（36文字）。個人利用APIなのでURLの美しさは優先度低。`google/uuid` を使う場合は外部依存が1つ増えるが、十分安定したライブラリ。
