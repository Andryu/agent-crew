# ADR: lessons ストレージ選定

**ステータス**: 決定済み  
**日付**: 2026-04-26  
**関連 Issue**: #83 (単体プロジェクト向け先行整備)、#82 (マルチプロダクト対応・将来)

---

## 決定

`_lessons.json`（JSON ファイル）を現時点では維持する。

---

## 背景

`scripts/lessons.sh` が `~/.claude/_lessons.json` に教訓を蓄積している。スキーマ v1.2.0 では `scope` / `stack` フィールドを追加し、将来のマルチプロダクト対応（Issue #82）に備える。ストレージ方式の選定を記録しておく。

---

## 理由

- Claude エージェントが主体で読み書きするため、`jq` との親和性が高い
- 単体プロジェクト運用では `flock` 問題を `mkdir` ロックのフォールバックで対処可能（POSIX 保証のアトミック操作）
- 現時点で管理する lesson 件数は数十〜数百件程度であり、JSON で十分
- 外部サーバー・追加ランタイム不要で、個人開発環境での管理コストが低い

---

## 棄却した代替案

### SQLite WAL

**候補として有力だが現時点では見送り**

| メリット | デメリット（現時点） |
|----------|---------------------|
| 複数リーダー + 1ライターを並行処理（ACID 保証） | 単体プロジェクトでは過剰 |
| Python 標準ライブラリ `sqlite3` で追加依存なし | `jq` 直接操作ができなくなる |
| Mem0 / Cognee / Cloudflare Agent Memory での採用実績あり | エージェントのクエリ記述が複雑化 |

**再検討タイミング**: Issue #82（マルチプロダクト対応）着手時に改めて評価する。

### PostgreSQL

外部サーバーが必要で、個人開発での管理コストが高いため棄却。

### ベクトルDB（Mem0 式）

数十〜数百件規模には意味的検索は過剰。棄却。

---

## 今後の予定

- Issue #83 完了後、単体プロジェクトでの安定稼働を確認する
- 安定確認後に Issue #82 でマルチプロダクト対応を着手し、その時点で SQLite 移行を再評価する
