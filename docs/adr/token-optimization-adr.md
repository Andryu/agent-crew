# ADR-004: サブエージェントのトークン消費最適化とレート制限対策

## Status
Accepted

## Context

agent-crew でスプリント実行中にClaude APIのレート制限に到達する問題が発生した。
実測データ（Sprint-03実行時）：

- 1スプリント（12タスク）で8つのサブエージェントを起動
- 合計ログサイズ: 約1MB（大量のトークン消費）
- architectエージェント（read-heavy）: 5回全て正常完了
- engineer-goエージェント（write-heavy, Bash多用）: 2回とも内部エラー（ファイルは書き込み済み）
- レート制限で1回完全停止（11時リセットまで待機）

サブエージェントに大量のシステムプロンプト（エージェント定義全文）と参照ファイルが渡される構造が
主なトークン膨張の原因と考えられる。また、Bashツールを多用するエージェントはコンテキストが長くなり
タイムアウト・接続断のリスクが高い。

## Decision

### 1. サブエージェント委譲基準の明確化

#### サブエージェントに委譲する（委譲すべき）タスク
- 独立して完結するタスク（他のコンテキストを参照しない）
- 成果物が明確なファイル（ADR, 設計文書, コード）の生成
- read-heavy な調査・設計タスク（Alex, Hana, Kai, Ren）

#### 親エージェント（Yuki）が直接処理する（委譲しない）タスク
- キュー操作（`scripts/queue.sh`の呼び出し）
- Slack通知の送信
- 単一ファイルの軽微な修正（complexity: S かつ Bashツール不要）
- 進捗確認・ステータス集計

委譲判断の基準式：
```
委譲コスト（トークン） > 親処理コスト（トークン） → 親で処理
専門知識が必要 or ファイル数 > 3 → サブエージェントへ委譲
```

### 2. プロンプト最適化ルール

#### 渡すファイルの上限
- サブエージェント起動時に参照させるファイルは最大3ファイルまで
- エージェント定義ファイル（.claude/agents/*.md）は起動時に自動で読み込まれるため、
  重複してプロンプトに含めない

#### 依頼文の構造
```
# 依頼（最大200字）
[何を作るか、1〜2文]

# 入力
- 参照ファイル: [最大3件のパス]
- 制約: [必須事項のみ箇条書き、最大5項目]

# 出力先
- [ファイルパス]
```

#### 禁止パターン
- 仕様書全文をプロンプトに貼り付ける
- 「〇〇を参照してください」と複数ファイルを列挙する（エージェント自身に判断させる）
- 完了報告フォーマットをプロンプトに毎回含める（エージェント定義に記載済みのため）

### 3. Bash多用を避けるパターン

#### write-heavyエージェント（Riku相当）向けルール
- **Bashコマンド実行は1タスクあたり最大5回まで**を目安とする
- ファイル書き込みはBashではなくWriteツールで行う
- `git add / commit / push`は1つのBashコマンドにまとめる（チェーン実行）
- 長時間かかるコマンド（ビルド・テスト等）は実行前に親へ確認を求める

#### タイムアウトリスクの高い操作
以下はサブエージェントに委譲せず、親または手動で実施する：
- `go build` / `npm install` など依存取得を伴うコマンド
- 外部APIへの連続リクエスト
- `git push`（ネットワーク依存）

### 4. レート制限対策

#### タスクのバッチ実行戦略

**スプリント規模ごとの推奨バッチ分割：**

| スプリント規模 | タスク数 | 推奨バッチ数 | バッチ間隔 |
|--------------|---------|------------|---------|
| 小 (〜10pt)  | 〜6件   | 1バッチ     | 不要 |
| 中 (〜20pt)  | 〜10件  | 2バッチ     | 30分以上 |
| 大 (〜30pt)  | 〜12件  | 3バッチ     | 1時間以上 |

1バッチ = サブエージェント起動数3〜4件を上限とする。

#### トークン消費の事前見積もり

タスク開始前に以下の式で消費量を粗見積もりする：

```
推定トークン = Σ(タスクごとの推定消費)

タスク推定消費:
  complexity S  → 15,000 tokens
  complexity M  → 40,000 tokens
  complexity L  → 80,000 tokens

バッファ係数: × 1.5（実測ベース）
```

1スプリント合計が推定300,000 tokensを超える場合は2バッチ以上に分割する。

#### レート制限到達時のグレースフル対応

1. 現在処理中のタスクのステータスを確認する（`scripts/queue.sh show`）
2. `IN_PROGRESS`のタスクがあれば、成果物ファイルの存在を確認する
3. ファイルが存在する場合 → 後述のリカバリ手順へ
4. ファイルが存在しない場合 → タスクを`TODO`に戻してリセット
5. レート制限リセット時刻（通常1時間後）まで待機
6. 待機中にリトライ計画（次バッチの対象タスク）を立てる

### 5. サブエージェント安定化

#### タイムアウト対策

- 1回のサブエージェント呼び出しで完結しない作業は事前に分割する
- タスクの完了条件を「出力ファイルの存在」で定義し、エージェントが返答できなくても
  ファイルを確認することで完了を判定できるようにする
- Bashコマンドのタイムアウトは `timeout 30 <command>` で明示的に設定する（長時間処理を除く）

#### ファイル書き込み済みだが完了報告が返せなかった場合のリカバリ手順

```
# Step 1: 成果物ファイルの存在と内容を確認
ls -la <期待するファイルパス>
head -50 <期待するファイルパス>

# Step 2: 内容が妥当であればタスクを手動でDONEにする
scripts/queue.sh done <slug> <agent> "リカバリ: ファイル確認済み、手動クローズ"

# Step 3: 次タスクへのhandoffを実行
scripts/queue.sh handoff <next-slug> <next-agent>
```

リカバリ条件（以下を全て満たす場合に手動クローズを許可）：
- 出力ファイルが存在し、内容が完了条件を満たしている
- `events[]`に`start`イベントが記録されている
- retry_countが MAX_RETRY 未満である

## Consequences

### 良くなること
- サブエージェント1回あたりのトークン消費が20〜40%削減できると見込まれる
- write-heavyタスクの内部エラー発生率が低下する
- レート制限到達時に作業を失わず再開できる

### 悪くなること・トレードオフ
- 委譲判断のルールが増え、Yukiの判断コストが増加する
- バッチ分割によりスプリント完了までの実時間が延びる可能性がある
- 手動リカバリ手順はオーナーの介入を要する場合がある

### 今後の課題
- トークン消費量のログ収集（`_signals.jsonl`への記録）と傾向分析（Sprint-04候補 #23）
- サブエージェントごとの消費量プロファイリング
- Bashタイムアウト値をタスク種別ごとに設定するルール整備
