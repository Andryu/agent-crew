## 1. pm.md に追記する「委譲判断フロー」

### 追記箇所

`## 委譲ルール` セクションの末尾に以下のブロックを追加する。

### 追記内容

```markdown
### 委譲 vs 親処理 判断フロー（ADR-004 Section 1）

サブエージェントを起動する前に以下を確認し、委譲コストを見積もること。

#### 親エージェント（Yuki）が直接処理するケース（委譲しない）

- Slack 通知の送信
- 単一ファイルの軽微な修正（complexity: S かつ Bash ツール不要）
- 進捗確認・ステータス集計（`queue.sh show` の実行）
- キュー操作 — ただし **Bash ツールを持つサブエージェント**（Alex 等）は
  自己タスクの `done` / `handoff` を自己実行してよい（後述）

#### サブエージェントへ委譲するケース

- 独立して完結するタスク（他のコンテキストを参照しない）
- 成果物が明確なファイル（ADR、設計文書、コード）の生成
- read-heavy な調査・設計タスク（Alex、Hana、Kai、Ren）

#### 判断基準式

```
委譲コスト（トークン） > 親処理コスト（トークン） → 親で処理
専門知識が必要 or 変更ファイル数 > 3 → サブエージェントへ委譲
```

#### サブエージェント起動時のプロンプト上限ルール

- 参照ファイルは最大 **3 ファイル**まで
- エージェント定義ファイル（`.claude/agents/*.md`）は起動時自動読み込みのため
  プロンプトに重複して含めない
- 依頼文は最大 **200 字**を目安とする
- 禁止: 仕様書全文の貼り付け、複数ファイルの列挙による「参照してください」指示
```

### Bash 実行上限ルール（ADR-004）

- Bash コマンド実行は **1 タスクあたり最大 5 回**を目安とする

## 3. complexity 別トークン見積もり表の記載先

### 判断

**pm.md の既存「複雑度見積もり（complexity）」セクションに統合する。**

理由:
- pm.md はすでに complexity の定義を持っており、スプリント計画時の参照先として確立している
- CLAUDE.md に置くと全エージェントへのコンテキスト増加につながりトークンを浪費する
- Yuki がスプリント計画時にバッチ分割判断をする場面で使うため、pm.md が最適

### pm.md への追記箇所

`### スプリントキャパシティ` セクションの直後に以下を追加する。

### 追記内容

```markdown
### トークン消費見積もり（ADR-004 Section 4）

スプリント開始前に以下の式で消費量を粗見積もりし、バッチ分割の要否を判断する。

| complexity | 推定消費トークン |
|-----------|----------------|
| S         | 15,000 tokens  |
| M         | 40,000 tokens  |
| L         | 80,000 tokens  |

**バッファ係数: × 1.5**（実測ベース）

```
推定合計 = Σ(タスクごとの推定消費) × 1.5
```

推定合計が **300,000 tokens** を超える場合は 2 バッチ以上に分割する。

| スプリント規模 | タスク数 | 推奨バッチ数 | バッチ間隔 |
|--------------|---------|------------|---------|
| 小 (〜10pt)  | 〜6件   | 1 バッチ    | 不要 |
| 中 (〜20pt)  | 〜10件  | 2 バッチ    | 30 分以上 |
| 大 (〜30pt)  | 〜12件  | 3 バッチ    | 1 時間以上 |

1 バッチあたりのサブエージェント起動数は **最大 3〜4 件**を上限とする。
```

## 4. レート制限到達時のリカバリ手順の記載先

### 判断

**pm.md の「ブロック時の対応」セクションを拡張する形で追記する。**

理由:
- レート制限対応は Yuki（PM・オーケストレーター）の責務
- 個別エージェントは自分のタスクのリカバリを Yuki に任せる設計が適切
- CLAUDE.md への記載はシステムプロンプト全体の肥大化につながる

### pm.md への追記箇所

`## ブロック時の対応` セクションの末尾に以下を追加する。

### 追記内容

```markdown
### レート制限到達時のリカバリ手順（ADR-004 Section 4）

レート制限（429 / 529 エラー）が発生した場合：

1. `scripts/queue.sh show` で全タスクのステータスを確認する
2. `IN_PROGRESS` のタスクがあれば、成果物ファイルの存在を確認する
   ```bash
   ls -la <期待するファイルパス>
   head -50 <期待するファイルパス>
   ```
3. ファイルが存在し内容が妥当 → 手動クローズ（Step 4へ）
4. ファイルが存在しない → タスクを `TODO` に戻してリセット
5. レート制限リセット（通常 1 時間後）まで待機
6. 待機中に次バッチの対象タスクを選定する

**手動クローズの条件**（以下を全て満たす場合のみ）：
- 出力ファイルが存在し、内容が完了条件を満たしている
- `events[]` に `start` イベントが記録されている
- `retry_count` が `MAX_RETRY` 未満である

```bash
# 手動クローズ手順
scripts/queue.sh done <slug> <agent> "リカバリ: ファイル確認済み、手動クローズ"
scripts/queue.sh handoff <next-slug> <next-agent>
```
```

## 5. ADR-004 Section 1 の改訂案

### 変更の背景

Sprint-05 で `architect.md`（Alex）に `Bash` ツールが付与された。
これにより Alex は `scripts/queue.sh` を直接呼び出せるため、
「キュー操作は Yuki が直接処理する」という Section 1 の記述が実態と乖離した。

### 改訂対象

`docs/adr/token-optimization-adr.md` の `### 1. サブエージェント委譲基準の明確化` 内
「親エージェント（Yuki）が直接処理する（委譲しない）タスク」の箇条書き。

### 現行テキスト

```markdown
#### 親エージェント（Yuki）が直接処理する（委譲しない）タスク
- キュー操作（`scripts/queue.sh`の呼び出し）
- Slack通知の送信
- 単一ファイルの軽微な修正（complexity: S かつ Bashツール不要）
- 進捗確認・ステータス集計
```

### 改訂後テキスト

```markdown
#### 親エージェント（Yuki）が直接処理する（委譲しない）タスク
- Slack通知の送信
- 単一ファイルの軽微な修正（complexity: S かつ Bashツール不要）
- 進捗確認・ステータス集計
- キュー操作 — ただし以下の例外を設ける：

  **例外: Bash ツールを持つサブエージェントの自己タスク操作**
  Alex（architect）など Bash ツールが付与されたエージェントは、
  自己タスクの完了操作（`scripts/queue.sh done` / `handoff`）を
  自己実行してよい。Yuki への委譲は不要。

  Yuki が直接処理するキュー操作の範囲:
  - 他エージェントのタスクへの `handoff`（スプリント全体調整が必要な場合）
  - `parallel-handoff`（並列実行管理）
  - `block` / `retry` の判断と実行
```

### 変更の影響

- Alex が自分のタスクを `done` + `handoff` することで、Yuki の仲介トークンを削減できる
- Bash ツールを持たないエージェント（Hana / Ren / Kai 等）は引き続き Yuki 経由のキュー操作となる
- 将来 Bash ツールを付与する際は同様の例外を適用できる
