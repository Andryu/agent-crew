# Riku 自己レビューチェックリスト 設計仕様

- **Issue**: #40
- **設計者**: Alex
- **作成日**: 2026-04-22
- **対象**: engineer-go.md / engineer-vue.md / engineer-next.md

---

## 背景と目的

sprint-02 の QA フェーズで `phase2-intelligence-qa` と `phase4-antigravity-qa` の2件に MAJOR 指摘が発生した。
Sora が発見する前に Riku 自身が潰せた指摘を、実装完了直後の自己レビューステップで検出できるようにする。

**設計の方針**:
- Sora のチェックリストと「重複」はしない。Sora は外部視点（仕様との整合性・セキュリティ脆弱性・テスト実行）を担う。Riku は実装者視点（実装が自分で意図した通りになっているか）を担う。
- 「チェック漏れを防ぐ」ではなく「MAJOR 指摘を事前に潰す」にフォーカスする。
- 項目数を絞る。全部チェックしようとすると流し読みになるため、MAJORに直結する項目のみ列挙する。

---

## Sora の観点との役割分担

| 観点 | Sora（外部QA）| Riku（自己レビュー）|
|------|--------------|-------------------|
| 仕様との一致確認 | Alex の設計・ADR と照合 | タスク定義（slug/notes）と実装が一致しているか |
| テスト実行 | `go test` / `npm run test` を実際に叩く | テストを書いたかどうか・意図したケースを網羅したか |
| セキュリティ | SQLインジェクション・XSS・認証漏れ | センシティブ情報のハードコードが無いか |
| 設計整合性 | ADR・Minaのコンポーネント仕様との照合 | 自分が決めた方針（実装メモ）を実際に守れているか |
| コード品質 | 命名・DRY・単一責任 | TODO/FIXME を意図なく残していないか |

---

## チェックリスト構造: 2層設計

```
Layer 1: 共通チェックリスト（スタック非依存）
  - 全 Riku エージェントが使う基本 12 項目
  - 「実装完了の定義（DoD）」セクションのすぐ下に配置

Layer 2: スタック別チェック（各ファイルに差分追加）
  - Go: エラーハンドリング・goroutine・インターフェース境界
  - Vue3: Props型・Pinia使用方針・scoped CSS
  - Next.js: Server/Client 境界・fetch キャッシュ制御
```

---

## Layer 1: 共通チェックリスト（全スタック共通）

実装完了後、Sora への handoff 前に必ず確認する。

```markdown
### 自己レビュー（Sora handoff 前に必ず確認）

#### 仕様確認
- [ ] タスクの `notes`（または設計ドキュメント）に書かれた全要件を実装した
- [ ] 実装していない要件が残っている場合、TODO コメントと理由を書いた

#### テスト
- [ ] 正常系・異常系それぞれのテストケースが存在する
- [ ] 境界値（空文字・ゼロ値・上限値）のテストがある
- [ ] テストを実際に実行して全件パスを確認した

#### コード品質
- [ ] エラーを握り潰している箇所がない（`_` で無視・空の catch ブロック等）
- [ ] センシティブ情報（パスワード・トークン・秘密鍵）がコード内にハードコードされていない
- [ ] 新規追加したパブリック関数・コンポーネントにコメントがある
- [ ] TODO/FIXME を残した場合は必ず理由をコメントに書いた

#### ビルド確認
- [ ] ビルドコマンドを実行してエラー・警告がゼロであることを確認した

#### handoff 準備
- [ ] 完了報告フォーマット（変更ファイル一覧 / 動作確認 / 特記事項）を記入した
- [ ] Sora が確認すべき「特記事項」（設計からの逸脱・未解決の懸念）を明示した
```

**項目数: 12項目**
意図的にこの数に抑える。項目が増えると形骸化するため、MAJORに直結しない項目はここには入れない。

## Layer 2: スタック別チェックリスト

### Go（engineer-go.md 用）

共通チェックリストの直後に追記する。

```markdown
#### Go 固有チェック
- [ ] エラーは `fmt.Errorf("context: %w", err)` でラップして上位に伝播している
- [ ] goroutine を起動した場合、終了の責任（`WaitGroup` / `context` キャンセル）を明示した
- [ ] 外部依存（DB・HTTP・ファイル）はインターフェース経由でモック可能な構造になっている
- [ ] `context.Context` を第一引数に受け取っていない関数を新規追加していない
```

### Vue3（engineer-vue.md 用）

```markdown
#### Vue3 固有チェック
- [ ] `defineProps` / `defineEmits` に型を明示した（`any` を使っていない）
- [ ] グローバル状態を Pinia で管理し、コンポーネントに直接 `ref` で持たせていない
- [ ] `<style scoped>` を使っており、グローバルスタイルへの影響がない
- [ ] API レスポンスを zod でバリデーションしている
```

### Next.js（engineer-next.md 用）

```markdown
#### Next.js 固有チェック
- [ ] `'use client'` の付与範囲が必要最小限（葉ノードのみ）になっている
- [ ] データフェッチは Server Component で行い、Client Component にはデータのみ props で渡している
- [ ] `fetch()` にキャッシュ制御（`cache` または `revalidate`）を明示した
- [ ] `any` を使わず TypeScript strict モードのエラーが 0 件である
```

## engineer-*.md への追記指針

### 追記位置

各ファイルの `### 実装完了の定義（DoD）` セクションと `### 完了報告フォーマット` セクションの間に挿入する。

```
[現在の構造]
### 実装完了の定義（DoD）
  - [ ] 機能が仕様通りに動作する
  - [ ] テストが通る
  ...

### 完了報告フォーマット
  ## 実装完了
  ...

[追記後の構造]
### 実装完了の定義（DoD）
  - [ ] 機能が仕様通りに動作する
  - [ ] テストが通る
  ...

### 自己レビュー（Sora handoff 前に必ず確認）
  #### 仕様確認
  ...（共通 12 項目）

  #### [スタック名] 固有チェック
  ...（スタック別 4 項目）

### 完了報告フォーマット
  ## 実装完了
  ...
```

### 追記の形式ルール

1. 見出しレベルは `###`（既存の DoDと同じ第三レベル）
2. サブカテゴリは `####`（第四レベル）
3. 各項目は `- [ ]` 形式（既存 DoD と揃える）
4. 共通部分は3ファイルに完全に同じテキストを入れる（DRY違反だが、参照コストを下げるためあえて冗長にする）
5. スタック固有部分は対応ファイルにのみ追記

### 追記してはいけないもの

- Soraのチェックリストと重複する観点（SQLインジェクション・XSS等のセキュリティ脆弱性）
- 「確認を推奨」レベルの MINOR 指摘に対応する項目（流し読みになるため）
- 実行コマンド（`go test ./...` など）—DoDセクションに既にある

## 設計トレードオフ

| 選択 | 採用した理由 | 諦めたもの |
|------|------------|-----------|
| 12項目に絞る | 項目が多いと流し読みになりMAJOR見落としが増える | MINOR指摘の事前防止 |
| DoDセクションの直後に配置 | 実装完了の確認の延長として自然に読まれる | 「完了報告フォーマット」の前に必ず目に入る配置 |
| 3ファイルに冗長にコピー | 各エージェントが使うファイルを単独で参照するため | ドキュメント管理コストの低減 |
| Soraの観点と明確に分離 | 役割の重複がチェック漏れの責任を曖昧にする | Rikuが「Soraが見るから省略」とならない前提 |
