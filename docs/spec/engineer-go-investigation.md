## 3. 根本原因の仮説

### 仮説 A（最有力）: サブエージェントのコンテキストウィンドウ超過

**メカニズム**:
1. 親から渡されるプロンプト = 「タスク説明」+ 「engineer-go.md 定義（313行）」+ 「参照ファイル群」
2. engineer-go は Bash/Read/Glob を多用するため、ツール呼び出し結果がコンテキストに蓄積する
3. queue.sh のような大きなファイル（実測 1000行超）を Read すると一気にコンテキストが膨張する
4. コンテキストが上限に近づくと Claude API 側が internal error を返してセッションを終了する

**根拠**:
- ADR-004 に「サブエージェントに大量のシステムプロンプト（エージェント定義全文）と参照ファイルが渡される構造が主なトークン膨張の原因」と記録
- 停止時はいずれも「queue.sh 全体を参照する指示」が含まれていた（Sprint-08 signals-impl）
- ファイルが書き込み済みだったケースがある＝実行中のある時点で超過した

### 仮説 B: タイムアウト（Claude API の応答時間上限）

**メカニズム**:
1. write-heavy タスク（多数の Bash + Write + Edit）は実行時間が長い
2. Claude API の内部処理タイムアウト（推定数分〜10分程度）に到達する
3. `[Tool result missing due to internal error]` はこの場合も発生する

**根拠**:
- token-optimization-adr.md で「Bashツールを多用するエージェントはコンテキストが長くなりタイムアウト・接続断のリスクが高い」と記録
- Sprint-05 でスマホ Remote Control スリープによる接続切断で Sora が停止した類似事象あり

### 仮説 C: レート制限との複合

**メカニズム**:
1. API 呼び出し回数がレート制限上限に近い状態でサブエージェントを起動する
2. 内部でのツール呼び出し（Read/Bash）のたびに API が呼ばれ、制限到達でエラーになる

**根拠**:
- Sprint-11 で Riku（engineer-go 相当）がレート制限中断を起こしている
- Sprint-03 でも「レート制限で1回完全停止」が記録されている

### 仮説 D: 無限ループ

**メカニズム**:
- エラーハンドリングのリトライロジックが再帰的に呼ばれ、ツール呼び出しが連続する

**評価**: 可能性は低い。engineer-go.md にリトライループは定義されていない。ただし Bash コマンドがエラーを返し続けた場合に連続実行する可能性はゼロではない。

## 4. 再現手順（仮説 A の検証）

以下の条件を意図的に作ることで再現できる可能性がある。

```
1. engineer-go への実装指示に以下を全て含める:
   - queue.sh 全文（1000行超）を参照させる指示
   - 設計書全文（500行以上）
   - 実装対象ファイルの既存コード全文
   - 「既存の全テストを実行して確認すること」という指示

2. 指示文字数が 3,000 トークン以上になるようにする

3. 実装タスクに Bash コマンドが 3 回以上含まれる構造にする
   （go test ./... など長時間コマンドを含める）
```

**注意**: 意図的な再現はコスト（トークン消費）が高いため、本番スプリントには含めない。

## 5. 検出方法

### 5-1. 事前検出（停止前に警告）

| 指標 | しきい値 | 対応 |
|------|----------|------|
| 親からの指示トークン数 | 2,000 tokens 超 | 分割を検討 |
| 参照ファイル数 | 4 件以上 | 3 件以下に絞る |
| 参照ファイルの合計行数 | 500行超 | 抜粋のみ渡す |
| complexity | L（大規模） | 2 つの M に分割 |

### 5-2. 事後検出（停止の確認）

```bash
# 1. キューで IN_PROGRESS のまま長時間止まっているタスクを確認
jq '.tasks[] | select(.status == "IN_PROGRESS")' .claude/_queue.json

# 2. 成果物ファイルの存在確認（書き込み済みかどうか）
ls -la <期待するファイルパス>

# 3. events に start はあるが done がない状態を確認
jq '.tasks[] | select(.status == "IN_PROGRESS") | .events' .claude/_queue.json
```

### 5-3. 停止パターンの分類

| パターン | ファイルの状態 | 推定原因 | 対応 |
|----------|--------------|---------|------|
| ファイルなし・エラー即時 | 存在しない | コンテキスト超過（初期段階） | タスク分割して再実行 |
| ファイルあり・途中で停止 | 部分的に書き込み済み | コンテキスト超過（実行中） | ファイルを確認し続きを別タスクで |
| ファイルなし・長時間経過後 | 存在しない | タイムアウト | 指示を短くして再実行 |

## 6. 対処方針

### 6-1. 即時適用（ルール追加）

**対象**: `engineer-go.md` の「実装ワークフロー」セクション

追記すべきルール:
```
### 指示受け取り時の確認事項

- 参照するファイルは最大 3 件。3 件を超える場合は実装に必要な箇所のみ Grep で抽出する
- 1 回の Bash 実行で長時間かかるコマンド（go test ./... / npm run build）は、
  それだけで 1 タスク相当のコストと見なす
- Read で 200 行を超えるファイルを読む場合は limit/offset を使って必要箇所のみ取得する
```

**対象**: 親エージェント（Yuki）の委譲指示テンプレート（pm.md / pm-protocol.md）

追記すべきルール:
```
### engineer-go 委譲時のチェックリスト

- [ ] 指示文が 2,000 トークン以下か（おおよそ A4 1ページ程度）
- [ ] 参照ファイルが 3 件以下か
- [ ] 大きなファイル（200行超）を丸ごと参照させていないか
- [ ] L タスクを渡していないか（M 以下に分割済みか）
```

### 6-2. アーキテクチャ対処（中期）

#### 案 A: タスク分割基準の厳格化

複雑度 M のタスクを engineer-go に委譲する場合の上限を設ける。

```
委譲可能条件（AND）:
- 変更ファイル数 <= 3
- 指示トークン数 <= 2,000
- Bash 実行予定回数 <= 5
```

上記を超える場合は、タスクを「設計確認タスク」と「実装タスク」に分割してから委譲する。

#### 案 B: Watchdog パターン（将来実装）

```
1. Yuki が engineer-go を委譲後、タイムアウト（例: 10 分）をセット
2. タイムアウト経過後に成果物ファイルの存在を確認
3. ファイルが存在しない場合 → BLOCKED として記録し、Yuki が直接実装を検討
4. ファイルが存在する場合 → 内容確認後にリカバリ手順（ADR-004 §5）を実行
```

**実装コスト**: 高（タイムアウト管理ロジックが必要）。Case B は Sprint-13 以降の候補。

#### 案 C: engineer-go の分割（最終手段）

engineer-go.md を Go 専用（`engineer-go-backend.md`）と Vue3 専用（`engineer-go-frontend.md`）に分割し、1 ファイルあたりの定義量を半減させる。

**参照**: `docs/adr/multistack-design.md` で既に設計済み。

**判断**: まず案 A（分割基準厳格化）と 6-1（即時ルール）で軽減を試みる。再発すれば案 C を検討する。

### 6-3. リカバリ手順の整備

ADR-004 §5 に記載されているリカバリ手順を engineer-go.md にも転記する。

```markdown
## 停止からのリカバリ（Yuki 向け）

1. 成果物ファイルの存在確認: `ls -la <期待するファイルパス>`
2. 内容が妥当であれば手動クローズ:
   `python3 scripts/queue.py done <slug> Riku "リカバリ: ファイル確認済み、手動クローズ"`
3. 内容が不完全であれば、残り実装を別タスク（smaller-<slug>）として追加し再委譲する
```
