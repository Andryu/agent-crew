# 設計メモ: 複雑度見積もり S/M/L — phase1-complexity

## 概要

GitHub Issue #11 対応。タスク分解時に複雑度 (complexity) を S / M / L で見積もるルールを整備し、
スプリント計画のキャパシティ管理に活用できる状態にする。

---

## 1. 現状分析

### `_queue.json` の現状

`complexity` フィールドはすでにスキーマに存在し、sprint-02 の全タスクで使用されている。
値は `"S"` / `"M"` / `"L"` の文字列が使われているが、定義がどこにも明文化されていない。

### `queue.sh` の現状

`normalize_events_filter` でスキーマ正規化（events / retry_count / qa_result）は行っているが、
`complexity` フィールドの存在チェック・値バリデーションは一切実装されていない。

### `pm.md` の現状

タスク分解テンプレートのテーブルに `complexity` 列が存在しない。
見積もりルールの記述も皆無。

## 2. 複雑度の定義と判定基準

| 値 | ストーリーポイント | 定義 | 典型的なケース |
|----|---------|------|--------------|
| `S` | 1 | 単一ファイル変更・明確な仕様 | テキスト修正、設定値変更、既存関数への引数追加 |
| `M` | 3 | 複数ファイル変更・設計判断あり | 新機能追加（2〜5ファイル）、スキーマ変更を伴う実装、既存モジュールのリファクタリング |
| `L` | 5 | アーキテクチャ変更・複数コンポーネント横断 | 新エージェント追加、queue.sh の責務変更、外部サービス連携導入 |

### 判定フローチャート

```
変更ファイルが 1 つ？
  YES → 仕様が完全に明確か？
          YES → S
          NO  → M
  NO  → アーキテクチャ決定や複数コンポーネント間の契約変更を含むか？
          YES → L
          NO  → M
```

### 判定の補足ルール

- 迷ったら上位に倒す（S か M で迷えば M、M か L で迷えば L）
- 設計タスク (Alex) は常に M 以上を基準とする（単一ファイルのドキュメント追記のみ S）
- QA タスク (Sora) は対象実装の complexity に 1 段階下を基準とする（L の実装 → M の QA など）

```markdown
## 複雑度見積もり（complexity）

タスク分解時に各タスクの complexity を S / M / L で設定する。

| 値 | ポイント | 定義 |
|----|---------|------|
| `S` | 1 | 単一ファイル変更・明確な仕様 |
| `M` | 3 | 複数ファイル変更・設計判断あり |
| `L` | 5 | アーキテクチャ変更・複数コンポーネント横断 |

### 判定基準

- **S**: 変更が 1 ファイル以内かつ仕様が完全に明確。テキスト修正・設定変更・ドキュメント追記など。
- **M**: 変更が 2〜5 ファイル、または設計判断（インターフェース定義・データ構造の選択など）が生じる。
- **L**: アーキテクチャ変更、新コンポーネント導入、複数モジュール間の契約変更を伴う。

迷ったら上位に倒す（S か M → M、M か L → L）。

### スプリントキャパシティ

タスク計画時に合計ポイントを算出し、スプリントの想定稼働量と比較する：

| スプリント規模 | 推奨ポイント上限 |
|--------------|----------------|
| 小（週 1〜2 日稼働） | 〜10 pt |
| 中（週 3〜4 日稼働） | 〜20 pt |
| 大（フル稼働） | 〜30 pt |

スプリント計画案に合計ポイントを必ず表示する：

> **合計ポイント: 12 pt**（S×2=2 + M×2=6 + L×1=5）
```

## 4. `queue.sh` バリデーション改修

### 4-1. 改修方針

`normalize_events_filter` に complexity の正規化（未設定時のデフォルト値）を追加し、
`cmd_start` で complexity の値バリデーションを実施する。

**トレードオフの検討:**

| 選択肢 | メリット | デメリット |
|--------|---------|----------|
| A: `start` 時にバリデーション | 着手前に強制できる | 既存タスクに complexity がなくても作成・ハンドオフはできる |
| B: `done` 時にバリデーション | 完了前に強制できる | 後付けになるため intent が遅い |
| C: 専用コマンド `validate` | 任意タイミングで検査可能 | 強制力が弱い |
| **D: `start` 時 + 警告のみ** | 後方互換性を保ちながら周知できる | 強制力が最も弱い |

**採用: 選択肢 A（`start` 時にエラー終了）**

理由: `start` はエージェントが着手する最初のアクションであり、この時点で complexity が未設定または不正であれば着手を止めるのが最も実害が少ない。既存タスクはすでに全て complexity が設定されているため後方互換性の問題はない。

### 4-2. 具体的な変更箇所

**`normalize_events_filter` に complexity の正規化を追加:**

```bash
normalize_events_filter='
  .tasks |= map(
    if has("events") then . else . + {events: []} end
    | if has("retry_count") then . else . + {retry_count: 0} end
    | if has("qa_result") then . else . + {qa_result: null} end
    | if has("complexity") then . else . + {complexity: null} end
  )
'
```

**`cmd_start` に complexity バリデーションを追加（depends_on チェックの後）:**

```bash
# complexity バリデーション
local complexity
complexity=$(jq -r --arg s "$slug" '.tasks[] | select(.slug == $s) | .complexity // "null"' "$QUEUE_FILE")
case "$complexity" in
  S|M|L) ;;
  *) release_lock
     echo "ERROR: complexity must be S, M, or L for $slug (got: $complexity)" >&2
     exit 10
     ;;
esac
```

**追加する exit code:**

| コード | 意味 |
|--------|------|
| `10` | complexity 値が不正（S/M/L 以外または未設定） |

### 4-3. `show` コマンドへの complexity 表示追加

`cmd_show` の要約表示に `complexity` を追加する：

```bash
# 変更前
jq '.tasks | map({slug, status, assigned_to, qa_result: (.qa_result // null), retry_count: (.retry_count // 0)})' "$QUEUE_FILE"

# 変更後
jq '.tasks | map({slug, status, assigned_to, complexity: (.complexity // null), qa_result: (.qa_result // null), retry_count: (.retry_count // 0)})' "$QUEUE_FILE"
```

## 5. スプリントキャパシティ管理の方針

### 5-1. ポイント体系

```
S = 1 pt  （Small: 単一ファイル、明確な仕様）
M = 3 pt  （Medium: 複数ファイル、設計判断あり）
L = 5 pt  （Large: アーキテクチャ変更、複数コンポーネント）
```

フィボナッチ数列を意識しつつも、3値に絞ることで見積もりコストを最小化する。

### 5-2. キャパシティガイドライン

個人開発の実態に合わせたざっくりとした上限値。厳密な管理よりも「詰め込みすぎ」の検知が目的。

| 想定稼働 | 推奨上限 | 根拠 |
|---------|---------|------|
| 週 1〜2 日（副業・週末） | 10 pt | L×2 か M×3 程度が上限感 |
| 週 3〜4 日（主軸開発） | 20 pt | L×4 か M×6 程度 |
| フルタイム 1 週間 | 30 pt | L×6 程度 |

### 5-3. Yuki が使う計算ロジック（参考）

タスク一覧確定後、以下で合計ポイントを計算して表示する：

```
S の件数 × 1 + M の件数 × 3 + L の件数 × 5 = 合計 pt
```

これをスプリント計画案に明示し、オーナーが「Go」を出す前にキャパシティを意識できるようにする。

### 5-4. Phase 2 リトロスペクティブとの連携インターフェース

Phase 2-D（リトロスペクティブ機能）では、完了したタスクの complexity を集計して「計画ポイント vs 実際の所要時間」を振り返る想定。
そのために以下の情報が `_queue.json` に残っていることが必要：

- `complexity`: 計画時の見積もり値（S/M/L）
- `events[].ts` の start〜done の差分: 実際の所要時間の近似値
- `summary`: 完了サマリー

retro フォーマットで参照できるよう、これらのフィールドは削除・上書きせず保持する。

| ファイル | 変更内容 |

### 実装上の注意事項

2. `normalize_events_filter` は複数コマンドで共有されているため、complexity の正規化を追加しても既存コマンドの動作に影響しない
3. `pm.md` の変更は既存フォーマットへの追記のみ。既存セクションの削除・移動は行わない

## 7. 設計上のトレードオフ記録

### ポイント体系を 1/3/5 にした理由

**代替案との比較:**

| 案 | ポイント体系 | 却下理由 |
|----|------------|---------|
| A | 1/2/3 | 差が小さく「L=少し大変」程度の感覚になりがち |
| **B（採用）** | **1/3/5** | S と L の比が 1:5 で「明確に別物」という感覚が出る |
| C | 1/5/13 | フィボナッチ的だが個人開発では過剰な精度感 |

### complexity をオプション（null 許容）にしない理由

- すでに sprint-02 の全タスクで設定されており、null を残す理由がない
- `start` 時にバリデーションすることで「設定忘れ」を早期に検知できる
- Phase 2 リトロスペクティブで集計する際に null があると計算が複雑になる
