このファイルは `pm.md`（Yuki）が参照する補助定義です。

---

## 複雑度見積もり（complexity）

タスク分解時に各タスクの complexity を S / M / L で設定する。
タスクを `_queue.json` に追加する際は必ず `complexity` フィールドを付与すること。

| 値 | ポイント | 定義 |
|----|---------|------|
| `S` | 1 | 単一ファイル変更・明確な仕様 |
| `M` | 3 | 複数ファイル変更・設計判断あり |
| `L` | 5 | アーキテクチャ変更・複数コンポーネント横断 |

### 判定基準

- **S**: 変更が 1 ファイル以内かつ仕様が完全に明確。テキスト修正・設定変更・ドキュメント追記など。
- **M**: 変更が 2〜5 ファイル、または設計判断（インターフェース定義・データ構造の選択など）が生じる。
- **L**: アーキテクチャ変更、新コンポーネント導入、複数モジュール間の契約変更を伴う。

迷ったら上位に倒す（S か M → M、M か L → L）。

### 判定フロー

```
変更ファイルが 1 つ？
  YES → 仕様が完全に明確か？
          YES → S
          NO  → M
  NO  → アーキテクチャ決定や複数コンポーネント間の契約変更を含むか？
          YES → L
          NO  → M
```

### エージェント別補足ルール

- **Alex（設計）**: 単一ファイルのドキュメント追記のみ S。それ以外は M 以上を基準とする。
- **Sora（QA）**: 対象実装の complexity に 1 段階下を基準とする（L の実装 → M の QA など）。

## リスクレベル設定（risk_level）

タスク分解時に各タスクの `risk_level` を設定する。
設定しない場合は `complexity` から自動推論される（S→low / M→medium / L→high）。

| 値 | 定義 |
|----|------|
| `low` | 既存パターンの踏襲。失敗しても影響が限定的 |
| `medium` | 新しい統合・API境界の変更を含む |
| `high` | アーキテクチャ変更・外部依存の新規追加・セキュリティ関連 |

### 強制 high にすべきケース

- DBスキーマの破壊的変更
- 外部 Webhook / API の新規連携
- エージェント間プロトコル（queue.sh のスキーマ）変更
- 過去に2回以上ブロックされたパターンと同種のタスク

## スプリントキャパシティ

タスク計画時に合計ポイントを算出し、スプリントの想定稼働量と比較する：

| スプリント規模 | 推奨ポイント上限 |
|--------------|----------------|
| 小（週 1〜2 日稼働） | 〜10 pt |
| 中（週 3〜4 日稼働） | 〜20 pt |
| 大（フル稼働） | 〜30 pt |

スプリント計画案に合計ポイントを必ず表示する：

```
> **合計ポイント: [n] pt**（S×[s件]=? + M×[m件]=? + L×[l件]=?）
```

## トークン消費見積もり（ADR-004 Section 4）

スプリント開始前に以下の式で消費量を粗見積もりし、バッチ分割の要否を判断する。

| complexity | 推定消費トークン |
|-----------|----------------|
| S         | 15,000 tokens  |
| M         | 40,000 tokens  |
| L         | 80,000 tokens  |

**バッファ係数: × 1.5**（実測ベース）

```
推定合計 = Σ(タスクごとの推定消費) × 1.5
```

推定合計が **300,000 tokens** を超える場合は 2 バッチ以上に分割する。

| スプリント規模 | タスク数 | 推奨バッチ数 | バッチ間隔 |
|--------------|---------|------------|---------|
| 小 (〜10pt)  | 〜6件   | 1 バッチ    | 不要 |
| 中 (〜20pt)  | 〜10件  | 2 バッチ    | 30 分以上 |
| 大 (〜30pt)  | 〜12件  | 3 バッチ    | 1 時間以上 |

1 バッチあたりのサブエージェント起動数は **最大 3〜4 件**を上限とする。

## レート制限到達時のリカバリ手順（ADR-004 Section 4）

レート制限（429 / 529 エラー）が発生した場合：

1. `scripts/queue.sh show` で全タスクのステータスを確認する
2. `IN_PROGRESS` のタスクがあれば、成果物ファイルの存在を確認する
   ```bash
   ls -la <期待するファイルパス>
   head -50 <期待するファイルパス>
   ```
3. ファイルが存在し内容が妥当 → 手動クローズ（Step 4へ）
4. ファイルが存在しない → タスクを `TODO` に戻してリセット
5. レート制限リセット（通常 1 時間後）まで待機
6. 待機中に次バッチの対象タスクを選定する

**手動クローズの条件**（以下を全て満たす場合のみ）：
- 出力ファイルが存在し、内容が完了条件を満たしている
- `events[]` に `start` イベントが記録されている
- `retry_count` が `MAX_RETRY` 未満である

```bash
# 手動クローズ手順
scripts/queue.sh done <slug> <agent> "リカバリ: ファイル確認済み、手動クローズ"
scripts/queue.sh handoff <next-slug> <next-agent>
```
